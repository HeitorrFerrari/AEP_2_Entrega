## Architecture

Standard Spring Boot layered architecture: Controller → Service → Repository → Model.

**Request flow:** Controller validates HTTP input via Bean Validation → Service enforces business rules → Repository (Spring Data JPA) → H2 in-memory DB. Schema created on startup via `ddl-auto=create-drop`; seed data from `data.sql`.

**Key domain concepts:**
- `Solicitacao` — citizen urban request (potholes, lighting, etc.). Has a `protocolo` (auto-generated by `ProtocoloService`), `StatusSolicitacao` state machine, `Prioridade` with SLA days, and optional `cidadao` link.
- `Denuncia` — anonymous or identified complaint, similar structure to `Solicitacao`.
- `Historico` — audit log of every status transition on a `Solicitacao`.
- `Atendimento` — record of a staff member attending to a `Solicitacao`.
- `Dashboard/Relatorio` — read-only aggregations, no dedicated entity.

**Status machine (`StatusSolicitacao` enum):** Each enum constant implements `transicoesPossiveis()` to enforce valid transitions. `SolicitacaoService.atualizarStatus` calls `podeTranzicionarPara()` and throws `BusinessException` on invalid transition. Status final states: `RESOLVIDO`, `ENCERRADO`.

**SLA / overdue:** `Prioridade` enum carries `slaDias` (CRITICA=1, ALTA=3, MEDIA=7, BAIXA=15). `Solicitacao.isAtrasada()` compares `prazoAlvo` with today. Updating status on an overdue request requires `justificativaAtraso`.

**DTOs:** All request DTOs are Java records with Bean Validation annotations. Response DTOs are plain records; `toResponse()` mapping lives in each Service (not a separate mapper).

**Exception handling:** `GlobalExceptionHandler` maps `ResourceNotFoundException` → 404, `BusinessException` → 422, validation errors → 400 using `ErroResponse`.

**Static analysis config:** Checkstyle uses `google_checks.xml`; PMD uses `quickstart.xml` ruleset. All tools configured with `failOnError=false` so build does not break on violations.
